为什么需要工厂模式？

DNEGEL3125 Lv1
  2025-03-12 00:45:18 2025-03-12 00:45:18 Created   2025-03-12 11:36:24 2025-03-12 11:36:24 Updated  
  • 设计模式 
 

想象一下你是一家玩具车组装厂的老板。你的工厂要生产各种各样的玩具车，比如：

  • 跑车 (SportsCar): 速度快，造型酷炫
  • 越野车 (OffRoadCar): 坚固耐用，适合崎岖地形
  • 卡车 (Truck): 载重能力强，适合运输货物

没有工厂模式的 “组装” 方式：

如果你的工厂没有使用工厂模式，那么每当客户想要一辆特定的玩具车时，你的组装工人（程序代码）就需要直接去组装。 例如，客户想要一辆跑车，工人就需要知道跑车的具体组装流程：

问题：

  1. 代码重复： 组装每种类型的车，都需要重复编写类似的组装步骤 (assembleEngine, assembleBody, assembleWheels, paint)，只是具体实现可能略有不同。
  2. 硬编码： 创建具体玩具车类型的代码 (new SportsCar(), new OffRoadCar()) 直接写死在 ToyCarAssembler 中。 如果要新增一种玩具车 (比如 豪华轿车)，就需要修改 ToyCarAssembler 的代码。
  3. 耦合性高： ToyCarAssembler 类直接依赖于 SportsCar, OffRoadCar 等具体玩具车类，修改任何一个玩具车类的实现都可能影响 ToyCarAssembler。
  4. 扩展性差： 增加新的玩具车类型很麻烦，需要修改 ToyCarAssembler 的代码，违反了 “开闭原则” (对扩展开放，对修改关闭)。

引入工厂模式的 “组装” 方式：

为了解决上述问题，我们可以引入 “玩具车工厂”。 工厂负责创建各种玩具车，而组装工人（ToyCarAssembler）只需要告诉工厂 想要什么类型的车，工厂就负责生产出来。

工厂模式的核心思想： 将对象的 创建 和 使用 分离。 将创建对象的职责交给专门的 “工厂” 类。

三种工厂模式及其应用场景：

现在我们来分别用玩具车工厂的例子，讲解三种工厂模式：

1. 简单工厂模式 (Simple Factory)

  • 概念： 一个工厂类 (比如 SimpleToyCarFactory) 包含一个静态方法 (或非静态方法)，根据客户端提供的参数 (比如玩具车类型字符串) 来创建并返回不同类型的玩具车对象。

  • 优点：

    • 封装了对象创建逻辑： 客户端代码不再需要知道具体玩具车类的创建细节，只需要跟工厂打交道。
    • 集中管理对象创建： 所有玩具车的创建都集中在 SimpleToyCarFactory 中，易于维护和管理。
    • 降低耦合： ToyCarAssembler 不再直接依赖于具体的玩具车类，只依赖于 ToyCar 接口和 SimpleToyCarFactory。

  • 缺点：

    • 违反开闭原则： 如果要新增一种玩具车类型 (例如 豪华轿车)，需要修改 SimpleToyCarFactory 的 createToyCar 方法，添加新的 if-else 分支。 工厂类的职责过重，变成了 “万能工厂”。

  • 应用场景：

    • 工厂类只负责创建少数几种相关的对象： 对象类型不多，变化频率不高。
    • 客户端只需要知道传入工厂的参数即可创建对象，不需要知道具体的创建过程。
    • 例如：
      • 日志记录器工厂: 根据日志级别 (debug, info, error) 创建不同的日志记录器对象。
      • 支付方式工厂: 根据支付类型 (支付宝, 微信, 银行卡) 创建不同的支付方式对象。
      • UI组件工厂: 根据组件类型 (按钮, 文本框, 标签) 创建不同的 UI 组件对象。

2. 工厂方法模式 (Factory Method)

  • 概念： 定义一个工厂接口 (或抽象类) ToyCarFactory，声明一个抽象的工厂方法 createToyCar(). 每个具体的玩具车类型 (或类型族) 都有一个具体的工厂类 (例如 SportsCarFactory, OffRoadCarFactory)，负责实现 createToyCar() 方法来创建对应类型的玩具车。

  • 优点：

    • 符合开闭原则： 如果要新增一种玩具车类型 (例如 豪华轿车)，只需要新增一个 LuxuryCarFactory 工厂类，实现 ToyCarFactory 接口，而不需要修改已有的工厂类或客户端代码。
    • 更灵活的创建逻辑： 每个具体工厂可以有自己的创建逻辑，可以根据需要定制化对象的创建过程。
    • 更好的扩展性： 易于扩展新的产品类型。

  • 缺点：

    • 类数量增加： 每种产品类型都需要一个对应的工厂类，导致类数量增加。
    • 客户端需要选择具体的工厂： 客户端代码需要知道要使用哪个具体的工厂类。

  • 应用场景：

    • 当创建对象的逻辑比较复杂，需要更灵活的创建过程时。
    • 当需要支持不断扩展新的产品类型时。
    • 当需要由子类决定创建哪个对象时 (工厂方法通常是抽象方法，由子类实现)。
    • 例如：
      • 图形库中的形状工厂: 每个形状 (圆形, 矩形, 三角形) 都有各自的工厂类来创建，可以定制化形状的创建参数。
      • 数据库连接工厂: 不同的数据库 (MySQL, PostgreSQL, Oracle) 有不同的连接工厂，负责创建特定数据库的连接对象。
      • 操作系统平台相关的组件工厂: 不同的操作系统 (Windows, macOS, Linux) 有不同的 UI 组件工厂，负责创建平台特定的 UI 组件。

3. 抽象工厂模式 (Abstract Factory)

  • 概念： 当需要创建 一组 相关的对象 (产品族) 时，可以使用抽象工厂模式。 抽象工厂接口 (或抽象类) 定义了一组创建相关产品的方法。 每个具体的工厂类负责创建 一整套 相关的产品。

  • 玩具车工厂例子：

    假设我们不仅要生产玩具车，还要生产玩具车的 零件，例如：

    • 引擎 (Engine): 分为高性能引擎和普通引擎
    • 车身 (Body): 分为运动型车身和越野型车身
    • 轮胎 (Wheels): 分为跑车轮胎和越野轮胎

    现在，我们希望创建不同 风格 的玩具车，例如 “运动风格” 和 “越野风格”。 运动风格的玩具车应该使用高性能引擎、运动型车身和跑车轮胎。 越野风格的玩具车应该使用普通引擎、越野型车身和越野轮胎。

    抽象工厂模式可以解决这个问题。

  • 优点：

    • 创建产品族： 可以保证创建出来的产品属于同一个产品族，具有一致性。 例如，运动风格的玩具车的所有零件都是运动风格的。
    • 易于切换产品族： 只需要更换不同的具体工厂，就可以切换到不同的产品族。 例如，从 SportsToyPartsFactory 切换到 OffRoadToyPartsFactory。
    • 隔离了具体产品的实现： 客户端代码不依赖于具体的产品类，只依赖于抽象工厂和抽象产品接口。

  • 缺点：

    • 难以扩展新的产品线： 如果要新增一个产品线 (例如，除了引擎、车身、轮胎，还要增加 “车窗”)，需要修改抽象工厂接口 ToyPartsFactory 和所有的具体工厂类，比较麻烦。

  • 应用场景：

    • 当需要创建一组相关或相互依赖的对象，并且这些对象构成一个产品族时。
    • 当需要在不同的产品族之间切换时。
    • 当需要保证产品族中对象的一致性时。
    • 例如：
      • UI 框架中的主题/风格切换： 不同的主题 (浅色主题、深色主题) 需要创建一套相关的 UI 组件 (按钮、文本框、菜单)，抽象工厂可以用于创建不同主题风格的 UI 组件。
      • 跨平台应用开发： 不同的操作系统 (Windows, macOS, iOS, Android) 有不同的 UI 组件和 API，抽象工厂可以用于创建平台相关的组件和 API 接口，实现跨平台兼容。
      • 游戏中的不同风格的美术资源： 不同的游戏风格 (科幻风格、魔幻风格、卡通风格) 需要创建一套相关的角色模型、场景模型、特效资源，抽象工厂可以用于创建不同风格的美术资源。

总结：

  • 简单工厂模式： 最简单的工厂模式，适用于创建对象类型不多，创建逻辑简单的场景。 主要优点是封装创建逻辑，降低耦合。 主要缺点是违反开闭原则，扩展性较差。

  • 工厂方法模式： 更灵活的工厂模式，适用于创建对象类型较多，创建逻辑复杂，需要扩展新的产品类型的场景。 主要优点是符合开闭原则，扩展性好。 主要缺点是类数量增加。

  • 抽象工厂模式： 更高级的工厂模式，适用于创建产品族，需要切换产品族的场景。 主要优点是创建产品族，易于切换产品族，保证产品族一致性。 主要缺点是难以扩展新的产品线。

选择哪种工厂模式取决于具体的应用场景和需求。 通常情况下，可以从简单工厂模式开始，如果需要更好的扩展性或更复杂的产品创建逻辑，再考虑工厂方法模式或抽象工厂模式。
